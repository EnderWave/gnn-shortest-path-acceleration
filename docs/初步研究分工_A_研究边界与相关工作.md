# 初步研究分工 A：研究边界、问题定义与相关工作

## 角色定位

A 负责第一轮初步研究的“方向盘”。

这次项目从零开始，不继承旧代码、旧样本构造路线和旧实现假设。A 的任务不是马上写程序，而是把研究问题压缩到一个清楚、可验证、不会和已有工作混淆的版本。

第一轮中，A 要保证所有人都围绕同一个问题工作：

> 只根据历史 OD 查询负载，选择值得精确压缩的连通区域；压缩表由经典精确算法计算；在线查询结果必须等于原图最短路。

## 第一轮核心判断

A 需要反复提醒团队：第一轮初步研究不是完整论文，也不是完整 GNN 系统。

第一轮只判断三件事：

1. 历史 OD 查询是否真的存在明显热点和结构性规律。
2. 在固定存储预算下，压缩部分连通区域是否有查询收益。
3. 不使用历史路径监督时，简单策略是否已经优于随机策略。

如果这三点不成立，后续直接上复杂模型意义不大。

## A 的主要任务

### 1. 重写项目边界

A 需要整理一份新的第一轮研究边界说明，明确：

- 输入是什么：图结构、边权、节点坐标、历史起点终点查询。
- 不输入什么：历史最短路径、历史边经过频率、历史节点经过频率、起终点对最短距离标签。
- 学习或策略模块做什么：选择压缩区域、估计区域价值、分配预算。
- 精确算法做什么：计算压缩表、执行查询、校验结果。
- 输出是什么：压缩索引、查询距离、查询统计指标。

这份说明要替代旧尝试中的“历史路径行为监督”思路。

### 2. 规定第一轮不做什么

第一轮明确不做：

- 不训练复杂 GNN。
- 不预测最短路径。
- 不预测最短距离。
- 不使用历史最短路径作为主监督。
- 不做多算法调度。
- 不做动态时变路网。
- 不追求大规模性能极限。

这些不是永远不做，而是第一轮先不做。

### 3. 梳理相关工作

A 需要读并整理相关工作，重点不是写长综述，而是找清楚本项目要避开的坑。

优先整理：

- workload-aware shortest path / distance querying；
- road network distance oracle；
- contraction hierarchies、hub labeling、tree decomposition；
- reinforcement learning for shortest path index；
- neural A*、learning-guided search；
- graph partitioning / graph coarsening 中和 shortest path 有关的工作。

每篇工作至少回答：

- 它解决什么问题。
- 它用什么输入。
- 它是否使用查询负载。
- 它是否使用路径或距离标签。
- 它是否保证精确性。
- 它的索引或预处理结构是什么。
- 它和本项目第一轮有什么区别。

### 4. 定义实验问题

A 要把第一轮实验问题写成可执行版本。

建议采用以下问题：

1. 热点 OD 负载相比随机负载是否更容易被索引压缩利用。
2. 随机连通区域压缩是否几乎没有稳定收益。
3. 只用起点/终点频率选择区域，是否优于随机区域。
4. 只用 OD 热点对关系选择区域，是否优于单点热度。
5. 使用历史路径频率的 oracle 能提供多大上界。

注意：oracle 可以用于对比，但必须标注它使用了路径信息，不属于主方法。

### 5. 统一术语

A 负责统一团队用词，避免三个人各说各的。

建议统一使用：

- 历史查询负载：历史 OD 记录。
- 无路径监督：不使用历史最短路径标签训练或选区。
- 连通区域：沿原图真实边连接的一组节点。
- 压缩表：区域内部或边界之间由精确算法计算出的距离信息。
- 存储预算：允许保存的区域、节点、边界距离表项数量。
- oracle 策略：使用历史路径信息的上界方法。

## 和 B 的接口

A 给 B：

- 第一轮必须统计的指标。
- 热点查询和冷门查询的定义。
- baseline 名单。
- 正确性校验要求。

B 给 A：

- 数据规模统计。
- 查询负载分布统计。
- baseline 运行结果。
- 哪些指标在工程上容易统计，哪些暂时困难。

## 和 C 的接口

A 给 C：

- 哪些区域选择策略属于无路径监督。
- 哪些只能算 oracle。
- 第一轮报告时如何描述压缩策略。
- 存储预算怎么写才合理。

C 给 A：

- 每种策略实际生成的区域特征。
- 索引收益和失败案例。
- oracle 与无路径监督策略之间的差距。

## 完成标准

A 的工作完成时，团队应该能清楚回答：

- 我们第一轮到底想证明什么。
- 为什么旧的路径监督尝试不能作为主线。
- 哪些方法是 baseline，哪些是 oracle。
- 如果第一轮结果不好，应该如何判断是方向问题还是实现问题。
