# 初步研究分工 C：从零实现压缩区域原型与简单策略

## 角色定位

C 负责第一轮初步研究的“方法原型”。

旧代码不再使用，因此 C 要从零实现一个最小压缩索引原型。第一轮不追求复杂模型，而是先判断压缩区域这个方向有没有基本收益。

C 的核心目标是：

> 在固定存储预算下，选择若干连通区域，物化精确距离信息，并验证它是否能减少查询开销且不破坏最短路正确性。

## 第一轮方法原则

第一轮只做可解释、可评测、可替换的简单策略。

优先级如下：

1. 区域必须连通。
2. 距离表必须由精确算法计算。
3. 查询结果必须正确。
4. 策略必须分清是否使用路径信息。
5. 先做简单策略，再考虑 GNN。

C 可以从零建立如下结构：

```text
src/
```

## C 的主要任务

### 1. 定义连通区域结构

C 需要设计一个简单区域对象，至少包含：

```text
region_id
nodes
internal_edges
boundary_nodes
seed_node
selection_method
storage_cost
```

区域要求：

- 节点集合在原图中连通。
- 内部边来自原图真实边。
- 能识别区域边界节点。
- 能估算存储成本。

第一轮建议先做不重叠区域，避免复杂冲突。

### 2. 实现区域生长

从零实现通用区域生长函数。

输入：

- 图。
- 种子节点。
- 最大区域大小。
- 扩展优先级。

输出：

- 一个连通区域。

第一轮至少支持：

- BFS 生长。
- 随机生长。
- 节点热度优先生长。

只要区域由邻接边一步步扩展，就能保证连通。

### 3. 实现存储预算

C 需要让区域选择受预算限制。

第一轮可以支持：

- 最大区域数量。
- 每个区域最大节点数。
- 总压缩节点数。
- 总边界距离表项数。

建议重点报告：

```text
storage_cost = sum(boundary_count * boundary_count)
```

这是一个粗略但直观的压缩表成本估计。

### 4. 实现无路径监督策略

第一轮至少实现两个不使用历史路径的策略。

#### 策略 1：随机连通区域

用途：最低 baseline。

方法：

- 随机选种子。
- 沿图随机或 BFS 生长。
- 在预算内生成区域。

#### 策略 2：起终点热点区域

用途：最简单无路径监督方法。

方法：

- 只统计节点作为 origin 的次数。
- 只统计节点作为 destination 的次数。
- 选择高频 OD 节点作为种子。
- 沿图生长区域。

#### 策略 3：热点 OD 对启发式区域

用途：利用起终点关系，但仍不使用路径。

方法可以很简单：

- 找出高频 origin-destination 热点对。
- 根据坐标或图距离近似选择源、目标附近区域。
- 在源热点、目标热点或两者之间候选位置生长区域。

第一轮不要求最优，只要求逻辑干净、能评测。

### 5. 实现 oracle 策略

C 可以实现一个使用历史路径频率的 oracle 策略，但必须清楚标注。

oracle 的用途：

- 不作为主方法。
- 不作为无路径监督结果。
- 只用于估计“如果知道历史路径，上界大概是多少”。

oracle 可以：

- 对训练查询运行 Dijkstra 获取路径。
- 统计节点或边经过频率。
- 选择高频路径区域压缩。

报告中必须写明：

> oracle 使用了历史最短路径信息，不满足无路径监督约束。

### 6. 构建压缩表

对每个选中区域，C 需要使用精确算法计算压缩信息。

第一轮建议：

- 找出区域边界节点。
- 在区域子图内，从每个边界节点运行 Dijkstra。
- 保存边界节点到边界节点的最短距离。

如果第一轮只评测距离查询，可以先不保存完整路径。

### 7. 实现混合查询

第一轮可以采用保守正确的混合图查询：

- 原图边保留。
- 区域边界距离表作为 shortcut 边加入查询图。
- 在线查询在混合图上运行 Dijkstra 或双向 Dijkstra。

由于 shortcut 距离来自原图中的精确最短路，所以不会产生比真实最短路更短的错误路径。最终仍要由 B 的校验函数确认。

## 和 A 的接口

C 从 A 获取：

- 哪些策略属于无路径监督。
- 哪些策略只能作为 oracle。
- 第一轮应该重点对比哪些方法。
- 结果报告中如何描述预算和正确性。

C 向 A 提供：

- 各策略区域示例。
- 区域统计。
- oracle 和无路径监督策略差距。
- 查询收益和失败案例。

## 和 B 的接口

C 从 B 获取：

- 图结构。
- 查询负载。
- Dijkstra 函数。
- 评测函数。
- 正确性校验函数。

C 向 B 提供：

- `build_index` 类接口。
- `query_with_index` 类接口。
- 索引预处理时间。
- 索引存储成本。
- 区域统计信息。

## 第一轮交付物

1. 连通区域数据结构。
2. 区域生长函数。
3. 随机区域策略。
4. 起终点热点区域策略。
5. 热点 OD 对启发式策略。
6. oracle 区域策略。
7. 区域边界距离表构建。
8. 混合查询原型。
9. 和 baseline 的对比结果。

## 完成标准

C 的工作完成时，应该做到：

- 每个区域都能验证连通。
- 每个策略都能在预算内生成区域。
- 压缩表能构建成功。
- 查询结果通过 B 的正确性校验。
- 随机、OD 热点、OD 对启发式、oracle 有清楚对比。
- 即使没有 GNN，也能判断是否值得继续研究学习式区域选择。

## 注意事项

- 不要复用旧代码。
- 不要一开始实现复杂神经网络。
- 不要生成不连通区域。
- 不要把 oracle 当主方法。
- 不要只做可视化，不做查询评测。
- 不要忽略存储预算。
