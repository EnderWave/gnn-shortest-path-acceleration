# 初步研究分工 B：从零搭建数据、查询器与评测框架

## 角色定位

B 负责第一轮初步研究的“实验底座”。

旧代码已经不再使用，所以 B 的任务是从零建立一个最小但可信的工程闭环。这个闭环不需要复杂，但必须干净、可复现、可比较。

B 的核心目标是：

> 给定一张图和一批 OD 查询，可以稳定运行精确最短路 baseline，统计查询开销，并为后续压缩索引方法提供统一评测接口。

## 第一轮工程原则

第一轮只追求最小可靠系统。

优先级如下：

1. 正确性优先。
2. 指标统一。
3. 代码结构清楚。
4. 小数据先跑通。
5. 后续再优化性能。

不要一开始就写复杂框架，也不要为了速度牺牲可检查性。

## 建议新建代码结构

B 可以从零建立如下结构：

```text
src/
  graph_io.py
  graph_types.py
  shortest_path.py
  workloads.py
  metrics.py
  evaluation.py

scripts/
  build_small_workload.py
  run_baselines.py
```

这只是建议结构，可以根据实现调整，但第一轮应保持职责清楚。

## B 的主要任务

### 1. 定义图数据结构

从零定义最小图结构，支持：

- 节点编号。
- 坐标，可选。
- 无向边或有向边。
- 正边权。
- 邻接表。

第一轮建议使用简单邻接表，不要过早封装复杂类。

需要支持的基本操作：

- 获取节点数量。
- 获取边数量。
- 获取某个节点的邻居。
- 检查节点是否存在。
- 检查边权是否为正。

### 2. 实现数据加载

B 负责实现或重新实现数据加载，不复用旧代码。

第一轮可以支持一种标准格式，例如 DIMACS 路网：

- `.co` 文件读取节点坐标。
- `.gr` 文件读取边。

也可以额外支持简单 CSV 边表，方便调试。

加载后需要输出数据摘要：

- 节点数。
- 边数。
- 是否有孤立点。
- 最大度数、平均度数。
- 边权最小值、最大值。

### 3. 生成或读取 OD 查询负载

B 需要提供一个干净的查询负载生成器。

第一轮至少支持三种负载：

1. 随机 OD 查询。
2. 热点到热点查询。
3. 热点到随机查询。

查询记录字段建议：

```text
query_id, origin, destination, query_type, count
```

如果需要热点信息，可以单独保存热点定义，不要把逻辑写死在评测代码里。

### 4. 实现精确 baseline

B 至少实现：

- Dijkstra。
- 双向 Dijkstra。

每个查询函数都要返回：

```text
distance
expanded_nodes
elapsed_ms
reachable
```

第一轮可以只返回距离，不必返回完整路径。路径返回可以后续作为调试或 oracle 构造使用。

### 5. 建立评测框架

B 负责写统一评测函数。

评测输入：

- 图。
- 查询列表。
- 查询方法。
- 方法名称。

评测输出：

- 平均查询时间。
- p50 查询时间。
- p95 查询时间。
- 平均展开节点数。
- 可达查询数。
- 不可达查询数。
- 正确率。

所有后续方法都必须通过这套评测函数，而不是各自统计一套指标。

### 6. 正确性校验

B 要提供统一正确性校验。

对于任何新方法，检查：

```text
method_distance(s, t) == dijkstra_distance(s, t)
```

如果权重是整数，使用严格相等。  
如果转成浮点，使用统一误差容忍。

正确性不通过的方法不能进入结果表。

## 和 A 的接口

B 从 A 获取：

- 第一轮需要回答的实验问题。
- 哪些指标必须报告。
- 哪些查询算热点，哪些算冷门。
- 哪些方法可以出现在正式对比表里。

B 向 A 提供：

- 数据摘要。
- 查询负载分布。
- baseline 结果。
- 正确性校验结果。

## 和 C 的接口

B 向 C 提供：

- 图结构和读取接口。
- 查询负载。
- baseline 查询器。
- 评测入口。
- 正确性校验函数。

C 向 B 提供：

- 索引构建函数。
- 索引查询函数。
- 索引存储统计。
- 预处理时间。

## 第一轮交付物

1. 干净的图加载模块。
2. 干净的查询负载生成模块。
3. Dijkstra 和双向 Dijkstra。
4. baseline 运行脚本。
5. 统一评测脚本。
6. 一份 baseline 结果表。

## 完成标准

B 的工作完成时，应该做到：

- 删除旧代码后，项目仍能从零跑出 baseline。
- 任意一个新压缩方法都能接入同一套评测。
- 查询距离正确性有自动校验。
- 热点查询和随机查询可以分开统计。
- 实验结果可以复现。

## 注意事项

- 不要复用旧代码。
- 不要一开始写模型。
- 不要把路径统计作为主流程的一部分。
- 不要只看时间，必须统计展开节点数。
- 不要为了快速演示跳过正确性校验。
